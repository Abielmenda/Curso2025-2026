Dataset Analysis for Museums of Madrid Explorer

1. Dataset Analysis

This section analyzes the dataset used for the Museums of Madrid Explorer application, focusing on its schema, quantities, and value ranges. The dataset is sourced from the Madrid City Council's open data portal and contains information about museums in Madrid.

Schema Analysis

The dataset is structured as RDF data in Turtle syntax, as shown in museums-madrid-example.ttl. It follows an ontology defined in museums-madrid.ttl. The main components of the schema are:

  • Classes:
    • mm:Museum: Represents a museum entity.
    • mm:Address: Represents the address of a museum.
    • mm:Barrio: Represents the neighborhood (barrio) of a museum.
    • mm:District: Represents the district of a museum.
  • Properties:
    • mm:name, mm:descriptionEntity, mm:schedule, mm:equipment, mm:transport, mm:description, mm:accessibility, mm:contentURL, mm:phone, mm:fax, mm:email, mm:type, mm:xCoordinate, mm:yCoordinate, mm:latitude, mm:longitude: Datatype properties for museum details.
    • mm:hasAddress, mm:hasBarrio, mm:hasDistrict: Object properties linking museums to their address, barrio, and district.
    • Address-related properties: mm:streetName, mm:streetClass, mm:numType, mm:num, mm:floor, mm:door, mm:stairs, mm:orientation, mm:locality, mm:province, mm:postalCode.
    • Barrio-related properties: mm:barrioCode, mm:barrioName.
    • District-related properties: mm:districtCode, mm:districtName.

Quantities and Value Ranges

The dataset contains entries for multiple museums, with two examples provided (Casa Museo Fuente del Rey and Casa Museo Lope de Vega). Key observations:

  • Number of Entries: The sample dataset includes two museum instances, but the full dataset likely contains more, covering major museums like Prado, Reina Sofía, and Thyssen-Bornemisza, as referenced in applicationRequirements.html.
  • Value Ranges:
    • mm:name: String, e.g., "Casa Museo Fuente del Rey", "Casa Museo Lope de Vega".
    • mm:descriptionEntity: String, detailed text describing the museum’s collection or history (100–500 words).
    • mm:schedule: String, human-readable schedules, e.g., "De lunes a viernes de 10:00 a 19:00 horas. Visitas guiadas concertadas Agosto cerrado".
    • mm:accessibility: String, values like "3" or "0,4", likely indicating accessibility levels (exact scale unclear without further documentation).
    • mm:latitude, mm:longitude: Double, e.g., 40.45546866305681, -3.7705716814130965, representing geographic coordinates in Madrid.
    • mm:xCoordinate, mm:yCoordinate: Integer, e.g., 434663, 4478596, likely UTM coordinates.
    • mm:postalCode: String, five-digit codes, e.g., "28023", "28014".
  • Data Completeness: Some fields like mm:equipment, mm:description, mm:floor, mm:door, mm:stairs, mm:orientation, and mm:fax are often empty, indicating optional or incomplete data.

2. Licensing Analysis

The dataset is sourced from the Madrid City Council’s open data portal, as referenced in datasetRequirements.html. The analysis of the licensing is as follows:

  • Publisher and Rightsholder: The Madrid City Council is the publisher and rightsholder of the dataset, as it is hosted on their official open data portal.
  • License: According to datasetRequirements.html, the dataset is released under an open license (e.g., CC-BY or public domain), which permits redistribution and publication for the Museums of Madrid Explorer application.
  • Proposed License for Generated Dataset: The generated Linked Data dataset will adopt the Creative Commons Attribution 4.0 International (CC-BY 4.0) license to ensure compatibility with the original dataset’s open license and to allow reuse while requiring attribution to the Madrid City Council.

3. Resource Naming Strategy

The resource naming strategy for the ontology and generated Linked Data is defined to ensure consistency, scalability, and compatibility with Linked Data principles. The strategy is as follows:

  • URI Domain: http://data.example.org/madrid/ will be used as the base domain for the ontology and data, following the example in museums-madrid-example.ttl.
  • Ontology URI: The ontology is defined at http://data.example.org/madrid/museos#, as seen in the provided museums-madrid.ttl.
  • Resource URI Pattern:
    • Museums: http://data.example.org/madrid/museos/resource/{museumID}, where {museumID} is a unique identifier (e.g., mmi:6017808 for Casa Museo Fuente del Rey).
    • Classes: http://data.example.org/madrid/museos#{className}, e.g., mm:Museum, mm:Address.
    • Properties: http://data.example.org/madrid/museos#{propertyName}, e.g., mm:name, mm:hasAddress.
  • Content Negotiation: The URIs will support content negotiation to deliver data in multiple formats:
    • RDF/Turtle (default): text/turtle
    • RDF/XML: application/rdf+xml
    • JSON-LD: application/ld+json
    • HTML: text/html for human-readable representations.
  • Path Structure:
    • Ontology: /madrid/museos#
    • Data instances: /madrid/museos/resource/{museumID}
  • Identifier Pattern: Museum identifiers will use the existing IDs from the dataset (e.g., 6017808, 4847190) to maintain consistency with the source data.
  • Reusability: The URIs are designed to be dereferenceable, allowing external systems to access and link to the museum data. The use of geo: properties (e.g., mm:latitude, mm:longitude) ensures compatibility with standard geospatial vocabularies.

Conclusion

The dataset satisfies the requirements for the Museums of Madrid Explorer application, with a well-defined schema and open licensing. The resource naming strategy ensures that the generated Linked Data is consistent, accessible, and reusable, aligning with the principles of Linked Data and the needs of the application.
